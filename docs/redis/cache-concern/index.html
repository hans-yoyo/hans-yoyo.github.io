Redis 缓存问题、淘汰策略

缓存穿透​

有一堆的请求，在缓存中没有，数据库中也没有，这种垃圾请求还特别多，而且因为是在数据库没有查询到，所以也不会被放到缓存中。大量的这种请求，最终会导致数据库压力剧增。

解决办法

  • 对请求参数进行校验
  • 缓存空值，对于空值是有限的，将没有命中的key，在缓存中设置空值，并设置过期时间；
  • 空值较多的情况，设置布隆过滤器

缓存击穿​

一个高频的热点数据，在失效的一瞬间，它的大量请求就会直接打到 DB 上，这样在 DB 还没有返回数据给 Redis 的时候，DB 承受了热点请求的压力，就好像缓存是一个水桶，然后突然水桶破了一个洞，直接冲垮了后面的堤坝（DB）。

解决办法

  • 对于热点数据，不要设置失效时间
  • 接口限流与熔断，降级。重要的接口一定要做好限流策略，防止用户恶意刷接口，同时要降级准备，当接口中的某些 服务 不可用时候，进行熔断，失败快速返回机制
  • 在第一个请求去查询数据库的时候对他加一个互斥锁，其余的查询请求都会被阻塞住，直到锁被释放，后面的线程进来发现已经有缓存了，就直接走缓存，从而保护数据库。但是也是由于它会阻塞其他的线程，此时系统吞吐量会下降。

缓存雪崩​

在同一个时刻批量过期的情况，因为有可能这些数据是批量插入的，所以他们的过期时间就会都在同一个时间。 正好在这个批量数据过期的时间点，大量的请求过来了，因为缓存数据过期了，所以没有命中缓存，直接请求到了数据库中。数据库的压力突然剧增，甚至有可能直接撑不住挂掉。然后有可能 DBA 会紧急重启 DB，但是刚一恢复，新的请求立马又把 DB 打垮了。 也有可能就是 Redis 挂了，缓存都不能用了，请求也是直接打到了 DB 上，然后 DB 也是扛不住压力，直接挂掉。再恢复，再挂掉。

和缓存击穿不同的是，缓存击穿指并发查同一条数据，缓存雪崩是不同数据都过期了，很多数据都查不到从而查数据库。

解决办法 事前

  • 如果是因为失效时间导致，在设置缓存的失效时间时，不要设置一样的时间，最好加个随机数
  • 缓存服务器集群部署，将热点数据均匀分布在不同的缓存数据库中

事中

  • 雪崩已经发生，考虑使用本地缓存，在雪崩发生时，本地缓存还能撑一会；
  • 另外使用Hystrix限流和降级

事后

  • 如果是缓存宕机了，开启缓存的持久化，保证服务器恢复后，缓存中的数据还在

布隆过滤器​

布隆过滤器，英文叫BloomFilter，可以说是一个二进制向量和一系列随机映射函数实现。 可以用于检索一个元素是否在一个集合中 通过布隆过滤器可以判断一个元素是否存在一个集合中，有可能存在，也可能不存在，因为存在碰撞，但如果判定不存在，则一定不存在。

总结

  • 布隆过滤器是用于判断一个元素是否在集合中。通过一个位数组和N个hash函数实现。
  • 优点：
    • 空间效率高，所占空间小。
    • 查询时间短。
  • 缺点：
    • 元素添加到集合中后，不能被删除。
    • 有一定的误判率

缓存过期策略​

定期删除​

Redis 会将设置了过期时间的 key 放到一个独立的字典里，以后会定期遍历这个字典来删除到期的 key。

Redis 默认100ms 扫描一次，过期扫描不会遍历过期字典中的所有 key

  1. 过期字典随机20个key
  2. 删除这 20 个 key 中已经过期的 key
  3. 如果过期的 key 的比率超过 1/4 ，那就重复步骤1

惰性删除​

惰性删除就是在客户端访问这个key时，检查key 的过期时间，如果过期则不会返回信息。

缓存淘汰策略​

不管是定期删除还是惰性删除，都不是一种完全精准的删除，还是会存在key没有被删除的情况，所以需要淘汰策略进行补充。

主要看分三类看：

1、不淘汰

noeviction （v4.0后默认的）

该策略是Redis的默认策略。在这种策略下，一旦缓存被写满了，再有写请求来时，Redis 不再提供服务，而是直接返回错误。这种策略不会淘汰数据，所以无法解决缓存污染问题。一般生产环境不建议使用。

2、对设置了过期时间的数据中进行淘汰

随机：volatile-random 在设置了过期时间的键值对中，进行随机删除

ttl：volatile-ttl 这种算法判断淘汰数据时参考的指标比随机删除时多进行一步过期时间的排序。Redis在筛选需删除的数据时，越早过期的数据越优先被选择

lru：volatile-lru LRU 算法的全称是 Least Recently Used，按照最近最少使用的原则来筛选数据

lfu：volatile-lfu

3、全部数据进行淘汰

随机：allkeys-random 使用 LRU 算法在所有数据中进行筛选。具体LFU算法跟上述 volatile-lru 中介绍的一致，只是筛选的数据范围是全部缓存，这里就不在重复。

lru：allkeys-lru

lfu：allkeys-lfu
